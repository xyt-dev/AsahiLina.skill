主要可以从以下几个角度分析：

---

## 建立地图

先建立地图：

```text
源码在哪些文件？
入口有哪些？分别在哪些文件第几行？
核心状态有哪些？分别在哪些文件第几行？
副作用边界在哪里真正发生？
你建议学生聚焦哪些文件学习？
```

## 中间层去噪

大型代码库的中间层层数多，通常是为了隔离变化维度。

阅读时先判断：

这一层是否真正改变状态？
这一层是否维护新不变量？
这一层是否触碰副作用边界？
这一层是否只是为了通用性 / 多平台 / 多后端 / 泛型灵活性？

## 状态机模型(阅读源码的重点)

把代码映射成状态机(以Rust为例作为参考)：

struct / enum = 状态载体
field = 状态变量
method / function = 状态操作
trait = 可替换的状态修改协议 / 行为的抽象契约
generic = 类型层面的变化维度
macro = 调用结构 / 状态机生成器
PhantomData = 类型系统状态标记
unsafe = 不变量担保区
cfg = 平台分支 / 平行宇宙
allocator / syscall / libc / IO / MMIO = 副作用边界

阅读时重点回答可以参考：

核心状态是什么？
状态何时被初始化和销毁？
状态存放在哪里？状态如何组织的？
状态之间有关联吗？是怎样关联和相互作用的？
哪些函数依赖哪些状态？
哪些函数改变哪些状态？
状态变化前后须满足什么条件？
哪些状态对外可见？哪些状态被封装？
错误路径会不会破坏状态？

## 不变量 / 契约模型

不变量 = 状态在整个生命周期内必须持续满足的契约

以Rust为例，可以参考：

模块对外承诺哪些不变量？
类型需要维护哪些不变量？
调用者保证哪些前置条件？被调用者保证哪些后置条件？
panic / error / early return 是否保证安全？
如果契约被撕毁，是 panic、错误返回、资源泄漏、逻辑错误，还是 UB？
